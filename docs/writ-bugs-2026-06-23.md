## Root-cause cluster: ONE global session-pointer file, N concurrent repos

**`/tmp/writ-current-session` is a single global file, written by every repo's
`UserPromptSubmit`.** With two Claude sessions open at once (Writ + mkt_engine), each
turn's hook clobbers the other's id. Last-writer-wins. This is the single root of B1 and
B2; the report's "two ids live simultaneously" and "rotation" are both this clobber.

**Live proof (captured this session):**

| Fact | Value |
|---|---|
| `/tmp/writ-current-session` | `1a5e5f50-…` |
| `1a5e5f50` actually belongs to | **mkt_engine** (`/tmp/writ-rag-debug.log`: `repo_root=…/mkt_engine port=9098`) |
| THIS (Writ) session id | `b6433218-…` (rag-debug `repo_root=…/Writ port=8783`) |

So while operating in Writ, the global pointer holds the *mkt_engine* session id. Any tool
that resolves "current session" by reading the file gets the wrong repo's session.

**Why daemon/graph isolation didn't cover this.** D4-02 (A-auto) isolates the *daemon
port* and the *graph socket* per repo, and the per-session cache files
(`writ-session-<sid>.json`) are already per-sid. But the **"current session" pointer was
never made per-repo** — it stayed a single `/tmp/writ-current-session`. It is the one
remaining global in an otherwise per-repo design.

- `.claude/hooks/auto-approve-gate.sh:55`

- **`skills/review-implementation/SKILL.md:55`** — `SID=$(cat /tmp/writ-current-session …)` then `--set-plan-reviewed` *(the B1 writer)*
- **`skills/subagent-driven-development/SKILL.md:104`** — same `cat` + `--set-plan-reviewed` *(the B1 writer)*

### Two different SID-resolution paths that disagree
- **Gate path** (`writ-sdd-review-order.sh:21`) → `detect_session_id "$PARSED"`
  (`bin/lib/common.sh:170`): envelope `agent_id` → `session_id` → PID → PWD-hash. This is
  per-session-correct (reads the live Task envelope).
- **Writer path** — the two SKILL.md call-sites above run
  `python3 "$WR/bin/lib/writ-session.py" update "$SID" --set-plan-reviewed "<task_id>" 2>/dev/null || true`,
  where `$SID` comes from the **global file** → may be the other repo's id.
- Result: write lands in `writ-session-<wrong-sid>.json`, gate reads
  `writ-session-<right-sid>.json` → looks empty → **deny** (B1).

### The writer line has THREE compounding flaws (codegraph + grep, 2026-06-23)
The exact command in both skills —
`python3 "$WR/…/writ-session.py" update "$SID" --set-plan-reviewed "<task_id>" 2>/dev/null || true`:
1. **`$SID` from the global file** → B1 (cross-repo clobber, above).
2. **Bare `python3`, not `$WRIT_PYTHON`.** Skills run outside hooks, so there is no
   `common.sh` `python3()` wrapper. On macOS `python3` = 3.9; `writ-session.py` has **no**
   `from __future__ import annotations` (verified) and uses `str | None` (e.g.
   `_log_friction_event`, `:16`) → **import crashes at load** (the documented py3.9 bug,
   WRIT-DEBUG-RECORDS B7 / STATE.md "py3.12 pin").
3. **`2>/dev/null || true` swallows that crash** → the setter silently no-ops,
   `review_ordering_state` is never written, and the gate denies forever with no error
   surfaced. (Explains "setter reported success yet gate denied" IF the agent's manual run
   used a ≥3.11 python while the skill-wired run used bare 3.9 — they take different python.)
